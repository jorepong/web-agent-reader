Researcher v2 architecture note

단일 재귀 에이전트 구조 점검

지금까지의 논의를 바탕으로, 하나의 Researcher가 자연어 입력과 자연어 출력을 유지하면서 루트, 발견 담당, 페이지 담당 상태를 어떻게 다룰 수 있는지 시각적으로 정리한 문서입니다.

합의된 중심 원칙

이 문서는 에이전트를 여러 종류로 나누는 대신, 하나의 Researcher가 호출 상태에 따라 다른 행동 집합을 받는 구조를 기준으로 합니다.

입출력은 자연어

외부 호출자와 부모 Researcher는 모두 자연어 task를 넘기고 자연어 report를 받습니다. 내부 구조체는 제어용 envelope일 뿐, 의미적 입출력은 자연어입니다.

task report same contract

하나의 에이전트

오케스트레이터, 탐색가, 합성가를 별도 페르소나로 만들지 않습니다. 같은 판단 원칙과 같은 보고 형식을 쓰는 단일 Researcher를 재귀 호출합니다.

recursive single prompt family

상태별 권한

같은 Researcher라도 열린 페이지가 있는지, 루트인지, 남은 예산이 있는지에 따라 선택 가능한 행동은 달라집니다. 이것은 다른 에이전트가 아니라 다른 실행 상태입니다.

schema gating runtime state

호출 상태 모델

루트와 서브 에이전트를 성격으로 구분하지 않고, startUrl과 depth 같은 호출 envelope로 구분합니다. 사용자가 보는 입력은 자연어이지만, 런타임은 제어 정보를 함께 들고 있어야 합니다.

페이지 없는 루트

사용자 질문을 처음 받은 상태입니다. 최상위 의사결정과 최종 응답을 책임집니다.

허용 delegate, delegate_parallel, done 비권장 직접 page read, 직접 search 결과 전문 누적 목표 작업 분해, 보고 비교, 최종 합성

페이지 없는 서브 에이전트

루트나 부모가 자연어 조사 작업만 넘긴 상태입니다. 출발점을 찾는 발견 담당입니다.

허용 search, map lookup, delegate, done 주의 깊은 페이지를 직접 여러 개 누적하지 않기 목표 좋은 시작점과 근거 있는 후보 만들기

페이지를 가진 서브 에이전트

startUrl을 받은 상태입니다. 하나의 investigation context를 책임집니다.

허용 read current page, explore links, done 제한 첫 행동 search 금지 또는 강한 제한 목표 해당 페이지와 연결 문맥에서 확인된 정보 보고 
runResearcher({
  task: "자연어 작업 지시",
  startUrl: undefined | "https://...",
  depth: 0 | 1 | 2,
  parentAgentId: null | "researcher-...",
  budget: sharedBudget
})

제안하는 재귀 흐름

루트가 직접 페이지를 읽지 않는 대신, URL 없는 서브 Researcher를 호출해 발견 작업을 맡길 수 있습니다.

두 설계안 비교

핵심 선택지는 루트가 직접 발견과 열람을 수행할지, 아니면 발견 작업까지 서브 Researcher에게 위임할지입니다.

루트가 직접 search/page surface를 여는 방식

구현은 단순합니다. 루트가 첫 검색을 수행하고 SERP나 지도 표면에서 링크를 골라 서브 에이전트를 부릅니다.

구현 단순성
루트 컨텍스트 보존
확장 표면 대응
책임 경계 명확성

루트가 발견도 delegate하는 방식

루트는 질문 분해와 최종 합성에 집중합니다. 검색과 출발점 발견은 URL 없는 서브 Researcher가 맡습니다.

구현 단순성
루트 컨텍스트 보존
확장 표면 대응
책임 경계 명확성
제 판단은 두 번째 방식이 장기적으로 더 안정적이라는 쪽입니다. 단, 루트가 처음부터 아무 판단도 못 하게 만들면 약해지므로 루트에는 page read가 아니라 작업 분해와 delegate 권한을 주는 것이 균형점입니다.

액션 설계 초안

자연어 입출력 원칙을 유지하면서도, 상태별 권한을 스키마로 제어하려면 액션을 다음처럼 나눌 수 있습니다.

루트 상태

사용자 질문 전체를 이해하고 조사 작업을 나눕니다. 페이지를 직접 열람하지 않습니다.

delegate delegate_parallel done

URL 없는 서브 상태

검색 또는 지도 조회로 시작점을 찾습니다. 발견한 후보를 더 깊은 서브 Researcher에게 넘길 수 있습니다.

search open_map delegate done

URL 있는 서브 상태

주어진 페이지를 먼저 읽습니다. 그 페이지의 링크 또는 직접 확인된 내용만 보고의 기반으로 삼습니다.

read_current explore explore_parallel done

예산 소진 또는 깊이 한도

더 이상 검색이나 하위 호출을 허용하지 않고, 지금까지 받은 자연어 보고로 답변을 마무리합니다.

done_only forced_synthesis

설계 판단 체크리스트

단일 에이전트의 정의

항상 같은 행동을 할 수 있다는 뜻이 아니라, 같은 판단 원칙과 같은 자연어 입출력 계약을 가진 재귀 단위라는 뜻으로 정의합니다.

루트의 책임

루트는 깊은 읽기를 하지 않고, 작업 분해, 서브 호출, 보고 비교, 최종 응답을 담당합니다. 이것이 루트 컨텍스트를 가장 깨끗하게 유지합니다.

자식의 책임

자식은 하나의 조사 문맥을 맡습니다. URL이 있으면 그 페이지를 먼저 책임지고, URL이 없으면 출발점 발견을 책임집니다.

자연어 보고 형식

보고는 자연어지만, 부모가 읽기 좋도록 ANSWER, SOURCES, COVERAGE, GAPS, NEXT_CANDIDATES 같은 섹션을 유지하는 것이 좋습니다.

스키마 제어

프롬프트로 행동을 부탁하는 것보다 상태별 스키마로 선택지를 줄이는 편이 안정적입니다. 특히 페이지를 받은 자식의 첫 행동 search는 스키마에서 빼는 편이 낫습니다.

생각해볼 질문

delegate는 필수 액션인가?

루트가 페이지를 직접 열지 않으려면 URL 없는 서브 Researcher를 호출할 수 있는 액션이 필요합니다. 이것이 없으면 루트가 첫 search를 해야 합니다.

search와 open_map은 같은 액션인가?

나중에 방문한 웹페이지 지도를 만든다면, 검색 엔진과 내부 지도는 모두 출발점 발견 도구입니다. 하나의 discover 액션으로 묶을지 분리할지 결정해야 합니다.

보고는 얼마나 구조화할 것인가?

입출력은 자연어로 유지하되, 섹션명을 엄격히 두면 부모 Researcher가 GAPS와 SOURCES를 더 안정적으로 해석할 수 있습니다.

자식이 search할 수 있는 순간은 언제인가?

페이지를 받은 자식은 먼저 페이지를 분석해야 합니다. search는 페이지와 링크가 실패했음을 보고한 뒤 부모가 새로 판단하게 하는 편이 더 깨끗할 수 있습니다.

이 문서는 구현 확정안이 아니라, Researcher v2의 재귀 구조를 점검하기 위한 사고 도구입니다.
